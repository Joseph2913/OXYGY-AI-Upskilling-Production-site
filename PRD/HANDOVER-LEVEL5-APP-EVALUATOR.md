## 1. What This Tool Is

The **AI App Evaluator** (Level 5 toolkit) evaluates an AI-powered application idea and produces a comprehensive product specification covering architecture, implementation, and risk assessment. It lives at:

- **Page component:** `components/app/toolkit/AppAppEvaluator.tsx`
- **Content/data:** `data/app-evaluator-content.ts`
- **Types:** `types.ts` (search for `AppEvaluatorResult`, `DesignScore`, `ArchitectureComponent`, `ImplementationStep`, `RiskItem`)
- **API hook:** `hooks/useAppEvaluatorApi.ts`
- **Backend (production):** `functions/src/index.ts` — `evaluateapp` Cloud Function
- **Route:** `/app/toolkit/ai-app-evaluator`

---

## 2. Current Architecture (2-Step Flow)

| Step | Name | What Happens |
|------|------|-------------|
| 1 | **Describe your application** | User fills 3 input fields (app description, problem/users, data/content) + example chips for quick-fill. Collapses to summary once output is generated. |
| 2 | **Review your evaluation** | 4-section output: Design Score (informational), Architecture & Components, Implementation Plan, Risks & Gaps. Collapsible sections, Cards/Markdown toggle, OutputActionsPanel, iterative refinement. |

**Step 2 shows a locked placeholder** when Step 1 isn't complete (uses `locked` prop on `StepCard`).

## 6. Key Design Decisions & Constraints

1. **Level accent colors for Level 5:** `LEVEL_ACCENT = '#C3D0F5'` (light violet), `LEVEL_ACCENT_DARK = '#2E3F8F'` (dark indigo)
2. **Cohesive deliverable:** `buildProductSpec()` combines Architecture + Implementation Plan + Risks. **Design Score is excluded** (informational only).
3. **Copy button label:** "Copy Product Specification" — never "Copy All"
4. **Refinement questions:** AI-generated via `result.refinement_questions` with 5 hardcoded fallbacks
5. **Font:** DM Sans only, inline styles, no CSS modules
6. **Layout:** `padding: '28px 36px'`, no maxWidth
7. **Firebase only** — no Vercel. Cloud Functions for backend
8. **OpenRouter only** — all AI calls via OpenRouter
9. **Rate limiting:** 1 request per 8 seconds (client-side, in `useAppEvaluatorApi`)
10. **OutputActionsPanel** uses `OutputActionsPanel` from `components/app/workflow/OutputActionsPanel.tsx` — shared across toolkit pages
11. **NextStepBanner** imported from `./NextStepBanner` — shared component added in cross-toolkit enhancement phase

## 7. API Response Shape

The `evaluateapp` Firebase function returns:

```typescript
{
  design_score: {
    overall_score: number;       // 0-100
    verdict: string;
    rationale: string;
    criteria: {
      user_clarity: { score: number; assessment: string };
      data_architecture: { score: number; assessment: string };
      personalisation: { score: number; assessment: string };
      technical_feasibility: { score: number; assessment: string };
      scalability: { score: number; assessment: string };
    };
  };
  architecture: {
    summary: string;
    components: Array<{
      name: string;
      description: string;
      tools: string[];
      level_connection: number;  // 1-5
      priority: 'essential' | 'recommended' | 'optional';
    }>;
  };
  implementation_plan: {
    summary: string;
    steps: Array<{
      phase: string;
      description: string;
      tasks: string[];
      duration_estimate: string;
      dependencies: string[];
    }>;
  };
  risks_and_gaps: {
    summary: string;
    items: Array<{
      name: string;
      severity: 'high' | 'medium' | 'low';
      description: string;
      mitigation: string;
    }>;
  };
  refinement_questions?: string[];  // 3-5 task-specific questions
}
```

**Note:** The `implementation_plan` field uses `.steps` in the TypeScript type but `getSectionSummary()` references `.phases` — this is a potential bug introduced in the post-modification phase (see Known Issues).

## 9. Known Issues / Future Work

1. **`getSectionSummary('implementation_plan')` references `result.implementation_plan.phases`** — but the `ImplementationStep` type uses `steps` not `phases`. This will throw a runtime error when the implementation_plan section is collapsed. Needs to be fixed to `result.implementation_plan.steps`.

2. **Dual save locations** — Save to Library exists in both the top action row AND the OutputActionsPanel. The Toolkit Page Standard (§4.2) says Save should only be in OutputActionsPanel. The current state has it in both places — this was a deliberate user/linter choice but may cause UX confusion.

3. **Dual download locations** — Similarly, Download exists in the top action row (`.md` only) AND in the OutputActionsPanel (`.md` + `.doc`). Could be streamlined.

4. **Design Review Output Standard not applied** — Unlike Level 2 (Agent Builder) and Level 4 (App Designer), Level 5 has not been updated to follow `DESIGN-REVIEW-OUTPUT-STANDARD.md`. The Quality Score Banner pattern (conic-gradient ring, expandable criteria) could replace the current `ScoreCircle` component for consistency.

5. **`architectureExpanded` and `planExpanded` state variables** — These are defined but no longer used (the `expandedSections` Set replaced them). They should be removed as dead code.

6. **Educational preview removed** — Step 2 no longer shows the educational default content (4-section preview) when Step 1 isn't complete. Instead it shows a locked placeholder. The educational preview code was in my initial write but the post-modification replaced it with the `locked` prop approach. The Toolkit Standard (§1.3, §1.4) says all steps should show educational content — this is a deliberate deviation.

7. **Example chips only show app name** — The standard (§4.6) says "Chip text should be complete enough to generate useful output on its own." Current chips show short names like "AI-Powered Student Portal" which populate all 3 fields. This works differently from single-field tools but achieves the same goal.

8. **Staggered animation count** — The `useEffect` for staggered blocks runs 6 iterations (for 4 sections + OutputActionsPanel + refinement card), but the sections each have their own `visibleBlocks` threshold check. Verify the timing feels right.
